## Context

Mid-pipeline, user flagged two missing gaps: (1) the built-in tool catalog was never
designed (spec only defines the `tools` registry *mechanism*, not the *catalog*; Unit 7
had a lone `read_file` demo tool); (2) — emphasized as 很重要 — the **permission model**:
who may call which tool, and how dangerous tools (bash/write/delete) are gated. Permission
is a control axis Alfred currently lacks, **orthogonal to `autonomy`** (autonomy = whether
auto-loops self-run; permission = whether a single tool call is allowed/asked/denied).

## Research Inputs (three references, user asked to study them)

- **Codex CLI** (codex.danielvaughan.com, developers.openai.com/codex): **two orthogonal
  axes** — `approval_policy` (untrusted / on-request / never / granular) × `sandbox_mode`
  (read-only / workspace-write / danger-full-access) + `network_access`; **both layers must
  agree** before an action runs. Named permission profiles (`-p`), command-prefix granular
  rules, optional auto-review subagent.
- **opencode** (opencode.ai/docs/permissions): **single axis, three states** — allow / ask
  / deny; **pattern-match, last-matching-rule-wins**; per-tool (read/edit/bash/webfetch) +
  **per-agent override** (merged with global, agent rules win); built-in `doom_loop` /
  `external_directory` default to ask. Example `bash: {"*":"ask","git *":"allow","rm *":"deny"}`.
- **Hermes** (github.com/NousResearch/hermes-agent): tiered RBAC (Owner/Admin/User/Guest)
  for gateway/messenger; per-skill ownership (`owner-write` default / `readonly`); skill-write
  approval gate; production isolation via Docker/Modal/Daytona sandbox backends.

## Decisions

| T1 | Which permission control model for Alfred? | **opencode-style single axis, three states (allow / ask / deny) + pattern-match + per-agent override**, as MVP. **Sandbox (Codex's capability-boundary axis) = TODO**, not MVP. | (1) Alfred is SDK-first, not a CLI that must self-sandbox — sandbox (Docker/Modal) is a deployment-time concern (Hermes only sandboxes in production); forcing it in MVP violates KISS. (2) opencode's per-agent override is isomorphic to Alfred's existing handoff tool-scope (#23a worker allowlist) — allow/ask/deny layers cleanly on top. (3) Three states simpler than Codex's two-axis, and `ask` composes with `autonomy`: off→ask becomes deny, assist→ask stays ask, auto→ask becomes allow. (4) pattern-match (`bash:{"rm *":"deny"}`) covers ~80% of dangerous-command risk at minimal cost without a sandbox. | user |
| T2 | Where does permission live in the build order? (architecture-debt correction) | **Permission is an L0 dispatch-level invariant, pulled forward INTO Tier-0** — woven into `_dispatch` at Unit 6 (registries + tool dispatch), NOT a late subsystem. | User: "一开始就该从 tool permission 设计,而不是 goal." Layering error — permission governs every tool call (foundation), goal is a Ring-3 plugin (L3). The original build order (spec §8) placed goal in the first post-Tier-0 wave but omitted permission entirely = built the 2nd floor before the foundation. Same class of gap as Eng-review L8 (autonomy gate must land before first auto-loop) — permission is the same-level foundation, missed then. Fail-loud + foundation-first → pull it forward. | user |
| T3 | How do the permission 3-states compose with the `autonomy` axis? | **Two orthogonal axes; `autonomy` only INTERPRETS `ask`, never overrides allow/deny.** Matrix: `allow`→run (all autonomy levels); `deny`→block (all levels, including `auto` — deny is a hard user wall even full-auto cannot break); `ask`→ off: treat as deny / assist: real interactive confirm (headless→deny) / auto: treat as allow. **Headless (cron/daemon, no TTY): `ask`→deny unless autonomy=auto**, with fail-loud log ("ask downgraded to deny: no interactive channel"). | Clean separation: `permission` sets the per-tool state, `autonomy` interprets only the single crossing point (`ask`). deny stays a true wall (auto can't break it); headless fails safe to deny. Zero responsibility overlap → no drift. | user |
| T4 | Built-in file + search tools: which mechanism? | **(a) File read/edit = hashline model built-in** (`hashread`/`hashedit`): every line tagged `LINE:HASH`; edits validate the content hash unchanged since read → prevents stale/torn edits. Replaces naive `read_file` + string-match edit. **(b) Search = `fff` built-in** (Fast File Finder: bigram index + mmap + SIMD + frecency-ranked LMDB → first result usually right). **fff is NOT a heavy dependency** — Alfred neither imports nor modifies it; it is a runtime executable invoked via subprocess (same class as ripgrep), zero added Python deps. **fff binary is bundled and shipped WITH Alfred** (user zero-install). Fallback chain on a platform where the bundled binary won't run: ripgrep → pure-Python grep, fail-loud WARNING. | hashline closes the "what I read == what I edit" invariant by mechanism, not by trusting the model to reproduce content (KB `llm-prompt-and-boundary-contracts` / `async-backend-contracts`); opencode/Hermes/Claude-Code all converging on it. fff: "better ranking beats faster" for agents (HN + entire.io evidence) — frecency one-shot retrieval saves tokens/turns. User: "fff 我们直接带着打包就行" — bundle it, not via mcp; the "heavy dependency" worry was unfounded (we don't change its code → it's just a binary + a data dir, like ripgrep). | user |
| T5 | How is the bundled fff binary packaged/distributed? | **(B) Per-platform companion packages** — main `agentkit` stays pure/small; the binary lives in `agentkit-fff-<platform>` (e.g. `agentkit-fff-linux-x64`, `-macos-arm64`, `-win-x64`) that the main package depends on per platform. Runtime: locate bundled fff → if absent/unsupported platform → fail-loud WARNING + fallback chain (T4). | Keeps core `agentkit` clean (SDK-first hygiene), isolates platform binaries in companion packages — the standard mature pattern (ripgrep-bin, node-sass). Not (A) platform wheels (more CI for the core wheel) nor (C) runtime-download (network dep + download-failure surface violates fail-loud startup determinism). | user |
| T6 | Built-in general tool catalog baseline + default permission per tool. | **Baseline = 7 built-in tools** (beyond the subsystem-owned ones already decided: `skills_list`/`skill_view`, `memory_append`/`memory_replace`/`memory_search`, `spawn_subagent`/`handoff_to`, `set_goal`, `edit_own_config`, mcp dynamic). The 7: `hashread` (read), `hashedit` (edit existing lines), `write_file` (create/overwrite), `fff` (search), `list_dir`, `bash`, `web_fetch` (see T7). **Tools stay separate, but write-permission is unified (opencode-proven):** `hashedit` + `write_file` (+ future `apply_patch`) share ONE `write` permission bucket (default **ask**); `hashread`/`fff`/`list_dir` = `read` (**allow**); `bash` = **ask** with `rm *`/`sudo *`/etc → **deny** patterns; `web_fetch` per T7. **Default-permission principle: read = allow, write/execute = ask, known-dangerous = deny.** **`hashread` + `hashedit` are a paired anchoring mechanism** (read emits `LINE:HASH|content`; edit anchors on that hash — same hashline system at both ends, mirroring oh-my-pi which overrides read+grep+edit together). NOT included: `delete_file` (dangerous → delete via `bash` + confirm), standalone `apply_patch` (covered by `hashedit`, rides the `write` bucket if added). | Tool-split vs permission-split are two separate questions (opencode resolves them oppositely): tools separate = clean LLM call semantics (hashline anchor is meaningful for editing existing lines, meaningless for creating an empty file); permission unified = one `write` gate covers all file mutation (opencode: "write tool is controlled by the edit permission, which covers all file modifications"). read-free / write-confirm / dangerous-blocked is safe-by-default under assist. Excluding delete/patch = YAGNI + Delete>Add. | user |
| T6a | hashline edit adoption confidence (evidence check). | **hashline (`hashread`/`hashedit`) confirmed as a validated baseline, not a frontier bet.** Default mutation surface in **oh-my-pi** (Can Bölük's Pi fork, 5100+ stars, ships it by default with a Rust engine + edit-reliability benchmark harness); plugin/proposal in opencode (`opencode-hashlines`), Hermes (#358), Claude Code (#25775). Benchmark lift: Grok Code Fast 1 edit success **6.7%→68.3%**, Claude Sonnet 4.5 +14.4%, Gemini +8%. Origin = Can Bölük's "The Harness Problem" — **same `harness>model` thesis as Alfred's premise**, so adopting it is thesis-consistent. | User wasn't sure which agent shipped it by default — verified: oh-my-pi. Order-of-magnitude benchmark + thesis-alignment turn this from "seems nice" into an evidence-backed default. | web + user |
| T7 | `web_fetch` — first outbound-network tool; permission + safety. | **Include `web_fetch` in the baseline, with a dedicated permission rule (NOT plain read=allow):** default **`ask`** (touching the network ≠ harmless local read); **hard `deny` patterns for SSRF/metadata:** `localhost`, `127.0.0.1`, `169.254.169.254` (cloud metadata), private ranges `10./192.168./172.16.` — **especially to protect the local key-proxy at `127.0.0.1:8888`** (fetching it = handing the agent its own API gateway/credentials). **Fetched content is marked untrusted** when injected into context (prompt-injection surface). Implementation: built-in via httpx (light dep, zero heavy deps); web fetch is a base capability that should ship in-box. | User flagged it 很重要. web_fetch breaks the previously-implicit local-only tool boundary — it is the first egress call, opening SSRF + injection surfaces. Network-only-read ≠ local read; deny on internal/metadata addresses closes the SSRF hole (the key-proxy is the sharpest risk); untrusted-marking aligns KB `llm-prompt-and-boundary-contracts` (untrusted-input surface). | user |
| T8 | (architecture confirmation, no change) Launcher vs kernel thickness. | **Confirmed: Alfred's existing launcher/kernel split is correct, no change.** Researched Hermes: its launcher (`cli.py` → `hermes_cli.main`) is a thin pure-dispatch wrapper, but `run_agent.py` behind it is a *substantive* kernel (closed learning loop, FTS5 memory, trajectory compression, subagent, skill self-improve); Docker uses s6-overlay supervising dashboard + gateways. **Lesson: "thin launcher" ≠ "thin kernel".** Alfred already mirrors this: `agentkit-cli`/`agentkit-server` = thin wrappers, `agentkit` = substantive kernel (spec §2). Unit 7's CLI must stay pure-dispatch / zero agent logic (already specified). | User intuition "Hermes 启动器就是个 wrapper?" — half right: launcher is thin, kernel is not. Recording so no one later削薄s the kernel because the launcher is thin. Confirms current design; produces no design change. | user + web |
| T9 | Permission SSoT + merge precedence across the 3 declaration sites. | **Three sites, merge toward strictest.** Sites: (1) `AgentConfig` permission block (global base) → (2) skill `allowed-tools` (narrows) → (3) per-agent / handoff tool-scope (#23a, narrows last). **Merge rule: each later layer can only narrow, never widen** (strictness lattice **deny > ask > allow**, take strictest); within a layer's pattern set, opencode **last-match-wins**. So all three sites can express policy but the result always converges toward safety. | Permission is multi-writer (cybernetics #4) — needs one convergence rule or it drifts. Monotonic-narrowing means no layer can escalate its own privileges (matches the #24 `autonomy`-self-edit ban spirit: lower layers can't grant themselves more). | user |
| T10 | `allowed-tools` advisory → enforced? | **Enforced.** A skill's `allowed-tools` now actually restricts which tools that skill may invoke (was MVP advisory-only in research). Folds into the T9 merge chain as the narrowing layer (2). | Now that a real permission model exists, advisory-only would be a silent no-op gate (open-loop, cybernetics #3). Enforcing it makes the declaration a real closed-loop constraint. | user |
| T11 | Cybernetics Lens applied (triggers: control-plane, multi-writer, SoT — all match). | Lens walked. System: Plant=tool-call side effects; Controller=permission rules (3 sites); Setpoint=only-authorized-ops-run; Disturbance=dangerous cmd / SSRF / agent misuse / headless no-confirm; Feedback=ask-confirm + deny-log + audit. **Most principles already satisfied by T1-T10** (#3 controllability=deny hard wall; #4 SoT=T9 strictest-merge; #5 hierarchy=autonomy×permission two layers). **One new finding (F-obs):** resolved permission is not observable — user can't debug "why can't the agent run this tool / which layer narrowed it". | Mandatory lens; report only new findings (S/N). | constraint |
| T11-F | Observability of resolved permission. | **Reuse the layered-instructions DEBUG manifest (L9-F1):** at session_start the `-v` manifest also lists, per tool, the **resolved permission (allow/ask/deny) + which layer set it** (config / skill / per-agent).
