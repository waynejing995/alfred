# Decision Log: Minimal Hermes-style Agent Loop

Started: 2026-06-15
Status: design-approved
Spec: ../specs/2026-06-15-alfred-design.md

## Context

Learning-oriented project: build a minimal, frontier-design agent loop from scratch.
Requested subsystems: cache, context, session, memory, skill, subagent, tool calling,
skill distill, dream, skill evolve, goal, cron task, mcp. Later: plugin. Plus a
**model fusion** feature (multiple models co-working on one task) and **multi-folder,
configurable skill loading**.

## Research Inputs

- Hermes Agent (Nous Research) — 5-pillar architecture (memory, skills, soul, crons,
  self-improvement); loop = "fills its own filing cabinet"; frozen-snapshot system
  prompt for cache; SQLite WAL + FTS5 sessions; progressive-disclosure skills (L0/1/2);
  self-registering tool registry; iteration budget; cron = fresh session per tick.
- KB insight `synthesis-harness-over-model`: harness > model is the dominant variable;
  simpler loops with strong models outperform complex scaffolds (mini-SWE-agent ~100
  lines >74% SWE-bench Verified). Implication: keep the core loop tiny.
- KB lesson `llm-prompt-and-boundary-contracts`: the LLM boundary is a high-attention
  finite channel + deploy-coupling surface + untrusted-input surface. Invariants in
  small system slot (<5KiB); validate model output with pydantic (extra=ignore at
  external boundary); prompt+parser are deploy-coupled.

| 7 | Which lifecycle events does the kernel emit? | Initial set, each with pydantic payload: `session_start`, `turn_start`, `pre_tool`/`post_tool`, `turn_end`, `session_end`, `idle`, `tick`/`job_due`, `skill_used`, `budget_warning`/`budget_exhausted`. | 8 | Kernel extension surface = how many registries? | Five symmetric registries: `tools` (name+schema+handler; used by user/mcp/plugin), `events` (lifecycle hook subscriptions), `models` (provider — used by model fusion), `skill_sources` (folder/source — used by multi-folder skill loading), `middleware` (optional loop-step interceptor). **plugin is not a 14th subsystem — it is the packaged form of calls into these registries.** Mechanism open (anyone may register), catalog converged (only these 5, all with real users — no speculative `formatters`/`parsers`/`storage_backends` tables). middleware kept (vs folded into pre/post_tool events) — TENTATIVE, revisit if events suffice. | Symmetric registries = the skeleton of an experiment bench; model-fusion's multi-provider and multi-folder skills need clean registration entry points, else they get hard-wired into core; YAGNI caps the table count. | user |
