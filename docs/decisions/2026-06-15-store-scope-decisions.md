## Context

Writing plan Milestone B (stores), user flagged Units 8-11 were too simplified — assumed a
single store location. Stores must be **scoped**: memory `core/` (persona/user) is global, but
session / trace / project-facts are per-project so one repo's conversation + learning doesn't
leak into another. User's initial framing was "store should live under the project". Research
revised this toward "global storage + project partition" (the mainstream pattern).

## Research Inputs

- **Claude Code**: global `~/.claude/projects/<project-path-hash>/sessions/<id>.jsonl` — sessions
  partitioned by hash of project ABSOLUTE PATH; isolated per project but physically under
  `~/.claude`, NOT in the project dir. Known pain: moving/renaming a project breaks the hash →
  session lost (issue #22387 requests in-project storage).
- **opencode**: global `~/.local/share/opencode/storage/session/<projectID>/...` — same pattern
  (global store + projectID partition). `OPENCODE_DATA_DIR` overridable. Community feature
  requests (#14292/#14959) ask for in-project / git-style storage — i.e. mainstream default is
  global-partitioned, in-project is the requested-but-not-default alternative.
- **Isomorphic to** layered-instructions (global vs per-tree) and the `ALFRED_HOME` single-root
  decision (L3).

## Decisions

| # | Question | Decision | Rationale | Source |
|---|----------|----------|-----------|--------|
| S1 | Store location model: global+partition vs in-project dir? | **(A) Global storage + per-project partition.** All under `~/.alfred/`, NOT in-project `./.alfred/` (user's initial idea B, revised by research). | (1) Doesn't pollute project dirs (B litters every repo with `.alfred/` + gitignore management); (2) no secret/session mis-commit risk (B's sessions.db could be committed); (3) consistent with the `ALFRED_HOME` single-root mental model (L3) — one global home, partitioned inside; (4) mainstream (Claude Code + opencode both do A). Trade-off: sessions don't follow the git repo — acceptable, agent conversation history is local machine state, not code. | user |
| S1a | Partition mechanism: per-project directories/files vs a DB column? **(corrects S1's implied per-dir layout)** | **Single DB + a `project_id` column + `WHERE project_id=?` filter** — NOT per-project directories or N db files. Default queries filter by current project; **cross-project search = drop the WHERE** (one-line SQL). Applies to session (`~/.alfred/sessions.db`), trace (`~/.alfred/trace.db` + JSONL bodies), and memory facts (`~/.alfred/memory/index.db`, facts rows carry `project_id`). | User: "如果是 db,是不是只要 db entry 带 project 就行?" — correct. Claude Code uses per-dir because its sessions are JSONL files (filesystem needs dir isolation); Alfred's session/trace/facts are SQLite (Unit 8/9/10), so isolation = a column. Benefits over per-dir: cross-project retrieval is trivial (toggle WHERE), ONE FTS5 index reused globally (not duplicated per project), one WAL connection pool, far simpler than dir-hash routing. | user |
| S2 | Which data is project-scoped vs global? (the global-vs-per-project SSoT, per store) | **Global (no `project_id`):** memory `core/persona.md` + `core/user.md` ("who the agent is / who the user is" — constant across projects; these ARE layered-instructions L2's two global layers). **Per-project (`project_id` column):** session, trace, memory `facts/`. | core = identity, cross-project invariant; facts/session/trace = "what happened / was learned in THIS repo", must not bleed across projects. Cleanly maps the global-vs-per-project intent onto presence/absence of the column. | user |
| S3 | What is `project_id`? | **(a) Normalized absolute path** of the project root (MVP). Readable, simple. **Move/rename loses the link** (Claude Code's known pain) — accepted for MVP; abstain from over-design. TODO seam: future stable project-id (a `.alfred-project-id` marker file with a generated uuid, or git-remote-based) + a `relink` command. | User chose (a). Path-as-id is the simplest workable key; the move-fragility is a real but low-frequency cost, and the fix (marker file / relink) is a clean future add, not MVP. Same discipline as deferring sandbox/worktree. | user |
| S4 | Facts retrieval scope (carried over from the Milestone-B open question). | **Default: current project only** (`WHERE project_id=?`). Because it's ONE index, "project + global merged retrieval" later = just widening the WHERE (zero structural cost — the mechanism is already there). | Per-project facts prevent cross-repo bleed by default; single-index design keeps the merge option free for the future without building it now (YAGNI + leave-the-seam). | user |
| S5 | Project-root discovery + no-project fallback (carried open question). | **Reuse layered-instructions L4a discovery:** walk cwd up to git root → that's the project root; no git root by `$HOME` → **cwd itself is the project root**. `project_id` = normalized path of that root. NO fallback to a shared "global/null project" (would mix all stray scripts' sessions into one bucket, defeating isolation). | ONE definition of "project" shared by instructions AND stores = single SSoT; user never hits "this AGENTS.md applied but my session isn't continuous" drift. cwd-as-project for non-git is deterministic and bounded. | user |
| S6 | DB substrate scope — **NARROWED (user pulled back from over-design).** | **DB only for retrieval-shaped / project-partitioned data: session, trace, memory facts.** Everything else uses the simplest medium: **skill content + versions = pure files** (`.versions/vN/` whole-dir snapshots incl. ref/script + `manifest.json` = active pointer + origin/parent/ts/lesson-bank/pass-rate); **goal = a small file** (`~/.alfred/goals/<thread_id>.json`, no `goals.db`); **distill high-water mark + proposal queue = persisted but NOT a new db** — reuse the existing `trace.db` (a couple of small meta tables), since they belong to the trace-consuming subsystem. revert = copy `.versions/vN/` back to active (file op). | User: "skill 放 db 会不会过度设计了" + "goal 用文件" + "distill 队列在 mem/db 都行" + "skill 回退到文件". Correct — for skill VERSIONS the managed object IS a file directory; a DB index over "which dir is active" adds a second SSoT (db-active vs file-active) and then needs S7/S8 machinery to repair the drift it created — over-design self-loop. session/trace/facts are genuinely DB-shaped (FTS5 + WHERE project_id). The earlier all-in-DB (old S6) bought SQL-queryable version history nobody needs (`ls .versions/` suffices) at the cost of drift. KISS + Delete>Add. | user |
| S7 | (SUPERSEDED by S6-narrowed) Active-version dual-declaration drift. | **No longer applicable — DELETED.** With skill versions back on pure files, there is only ONE declaration of active (the on-disk `SKILL.md` + `manifest.json` pointer); no DB `active_version` row exists, so the db-vs-file drift this decision resolved cannot occur. The loader was already file-only. | The drift S7 fixed was *created by* putting active in the DB; removing the DB ledger removes the problem at the source rather than patching it. | constraint |
| S8 | (SUPERSEDED by S6-narrowed) hash-as-detection for manual edits. | **No longer needed — DELETED.** File-based skills natively support hand-edits (the file IS the SSoT); a manual edit just is the new active, optionally noted in `manifest.json` history. No hash gate, no `origin=manual` reconciliation against a DB. `origin ∈ {human, distill, evolve, revert}` (drop `manual` — a hand-edit of a human skill is just `human`). | The hash machinery existed only to detect drift between DB-active and file-active; with no DB-active, there is nothing to detect. User's "don't foreclose manual edits" is satisfied trivially by files-as-SSoT. | constraint |
| S9 | Cybernetics Lens (re-checked after S6 narrowing). | Lens re-walked on the narrowed design. #4 SoT: each datum now has exactly ONE owner — session/trace/facts in their db (project_id column), skill content+version in files, goal in a file, distill cursors in trace.db. **No dual-SSoT remains** (the S7/S8 drift is gone by construction). #6 S/N + #7 min-effort: removing the skill-version DB ledger is a net deletion. No new findings. | Narrowing eliminated the only prior drift finding; the simpler design is strictly more lens-compliant (fewer writers, fewer mechanisms). | constraint |
